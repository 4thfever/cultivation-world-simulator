# 角色创建/删除后的刷新语义修复

当前状态：核心修复已落地。本文档记录角色创建与删除在前端的刷新语义，以及世界快照必须完整覆盖地图角色集合的约束。

## 1. 背景

前端现在的角色创建与角色删除，都走同一类“命令成功后再拉一次世界快照”的模式：

1. 创建角色：`useCreateAvatarPanel.handleCreateAvatar()` 在 `avatarApi.createAvatar()` 成功后调用 `worldStore.fetchState()`。
2. 删除角色：`useDeleteAvatarPanel.handleDeleteAvatar()` 在 `avatarApi.deleteAvatar()` 成功后，也调用 `worldStore.fetchState()`。

这条路径表面上是“同步世界状态”，但实际上会把两种不同语义混在一起：

1. 世界实体增量变化。
2. 事件栏与其他派生视图的全量重建。

结果就是：

1. 用户会感到创建/删除动作很慢，尤其在后台模拟器正在跑一轮 step 时，命令还要等同一把 mutation lock。
2. 事件栏会在刷新后突然清空，因为 `worldStore.fetchState()` 内部会调用 `applyStateSnapshot()`，而 `applyStateSnapshot()` 会直接 `eventStore.reset()`，但并不会顺手重新拉一页事件。

## 2. 已确认的问题范围

这个问题不只影响创建，也影响删除。

### 2.1 创建角色

创建流程的后端命令通过 `runtime.run_mutation()` 串行化，和 `sim.step()` 共用同一把锁。只要后台正在处理一次较慢的 step，创建请求就会排队等待。

前端在创建成功后又会调用 `worldStore.fetchState()`，进一步把 loading 时间拉长。

### 2.2 删除角色

删除流程和创建一样，也会在命令成功后调用 `worldStore.fetchState()`。

因此删除同样有两个副作用：

1. 命令本身可能因为和模拟 step 串行而等待。
2. 刷新后事件栏会被 `eventStore.reset()` 清空，且不会自动恢复。

所以删除不是一个独立问题，而是和创建共享同一套刷新语义缺陷。

## 3. 根因

### 3.1 前端把“全量世界快照刷新”当成了“命令后同步”的默认动作

`worldStore.fetchState()` 的目标是拉取完整世界状态快照，主要适合这些场景：

1. 开局初始化。
2. 读档或重载。
3. 需要把整棵前端状态树重新对齐的重型同步。

它不适合作为每次角色创建/删除后的默认同步方式。

### 3.2 `applyStateSnapshot()` 把事件视图当成了快照的一部分

`applyStateSnapshot()` 当前会：

1. 更新时间。
2. 覆盖 avatar 列表。
3. 清空事件 Store。
4. 更新天象与隐藏域。

但它没有重新加载事件分页，也没有把后端 `world/state` 已带回的 recent events 重新注入事件栏。

这就导致“快照刷新”把事件栏冲掉了。

### 3.3 创建/删除后的目标其实是“增量同步”

从业务上看，创建和删除命令成功后，前端真正需要的是：

1. 角色列表保持一致。
2. 世界时间、天象、宗门态势等必要派生状态保持一致。
3. 事件栏不要丢失已存在内容。

这更像是“局部补丁 + 必要查询”，而不是“整页重建”。

### 3.4 世界快照不得截断地图角色集合

`/api/v1/query/world/state` 是前端地图、角色筛选和初始化重建 `avatarStore` 的权威来源，因此其 `avatars` 必须覆盖当前所有存活角色。此前仅返回前 50 条角色会使任何位于截断窗口外的角色在页面刷新后同时从地图与事件筛选中消失。

## 4. 设计原则

1. 语义分层：命令结果、世界快照、事件分页分开处理，不要互相借壳。
2. 增量优先：创建/删除这种局部 mutation，优先用增量更新，不走重型全量刷新。
3. 事件保真：除初始化、读档、重置等明确的重建场景外，不允许普通命令把事件栏悄悄清空。
4. 统一入口：创建和删除采用同一套后处理策略，避免未来一个修了另一个漏掉。
5. 可诊断：如果命令等待了较久，前端应该更容易判断是“在等 mutation lock”还是“在等刷新请求”。

## 5. 目标行为

### 5.1 创建角色后的行为

1. 创建命令成功后，前端立即用返回的 `avatar` 摘要更新角色 Store。
2. 不再默认调用 `worldStore.fetchState()`。
3. 若确实需要同步少量派生状态，改为显式、最小化的查询。
4. 事件栏保持当前内容，不应被清空。

### 5.2 删除角色后的行为

1. 删除命令成功后，前端从角色 Store 中移除该角色，或通过最小化同步更新列表。
2. 不再默认调用 `worldStore.fetchState()`。
3. 事件栏保持当前内容，不应被清空。

### 5.3 初始化与重建场景的行为

1. 开局初始化、读档、显式重置等场景仍可以使用全量快照。
2. 这类场景下允许事件栏重建，但必须在逻辑上明确这是“重建”，而不是“普通创建/删除后的顺手刷新”。

## 6. 方案建议

### 6.1 前端命令后处理拆分

把“角色创建/删除后的刷新”从 `worldStore.fetchState()` 中拆出来，形成独立的命令后处理逻辑。

建议形态：

1. 创建成功后只更新 `avatarStore`。
2. 删除成功后只更新 `avatarStore`。
3. 必要时再按需刷新 `sectStore` 或其他局部派生数据。
4. 事件 Store 保持不动，除非明确触发事件页重载。
5. 创建成功后预加载新角色头像纹理；纹理缓存更新必须替换浅响应缓存对象，保证 Pixi 组件能从兜底图形切换到真实头像。

### 6.2 `worldStore.fetchState()` 只保留给重建型场景

`fetchState()` 应该继续存在，但语义更明确：

1. 适合初始化、读档、重置。
2. 不适合作为普通命令后的通用刷新手段。

如果未来还有需要“全量对齐”的命令，也应显式选择这条路径，而不是隐式复用。

### 6.3 事件栏的刷新由事件 Store 自己负责

事件栏是否要重载，应由事件面板或事件 Store 自己决定，而不是被世界快照的副作用触发。

这样可以保证：

1. 事件分页逻辑独立。
2. 事件筛选条件独立。
3. 世界快照刷新不会误伤事件视图。

### 6.4 创建/删除命令在等待状态上保持一致

如果前端希望在打开系统面板后避免“卡在当前 step 的尾巴上”，可在进入创建/删除相关操作前使用已有的 `pause-and-drain` 语义，让当前模拟步收尾再执行命令。

这不是必须和事件栏修复绑定在一起，但它是同一类体验问题的配套优化。

## 7. 实现计划

### 7.1 前端

1. 调整 `useCreateAvatarPanel.handleCreateAvatar()`，移除命令成功后的默认 `worldStore.fetchState()`。
2. 调整 `useDeleteAvatarPanel.handleDeleteAvatar()`，移除命令成功后的默认 `worldStore.fetchState()`。
3. 用命令返回值直接做 `avatarStore` 增量更新，删除则做局部移除。
4. 如确有需要，再用显式的最小化查询刷新局部派生状态。
5. 补测试，覆盖：
   - 创建成功后不再触发全量 world refresh。
   - 删除成功后不再触发全量 world refresh。
   - 事件 Store 不会因为创建/删除而被清空。

### 7.2 世界快照语义

1. 保持 `worldStore.initialize()` 可重建事件栏。
2. 保持 `worldStore.reset()` 可清空状态。
3. 明确普通 `fetchState()` 不应隐式破坏事件栏。

如果实现上仍需要保留 `applyStateSnapshot()` 的事件清空动作，应把它收口到初始化/重建路径，而不是普通刷新路径。

### 7.3 后端

后端创建与删除命令本身不需要改成返回整套世界状态。

当前更适合的方向是：

1. 保持命令只返回必要的实体摘要。
2. 让前端根据摘要做局部更新。
3. 把全量重建留给初始化与读档。
4. 世界快照返回所有当前角色，不得使用固定前缀截断来构造地图实体集合。

这样可以减少响应负担，也让命令语义更清楚。

## 8. 验收标准

### 8.1 功能验收

1. 创建角色后，角色能立即出现在前端列表里。
2. 删除角色后，角色能立即从前端列表里消失。
3. 创建/删除后，事件栏不会突然变空。
4. 创建/删除后，不再因为额外的全量 world refresh 造成明显长等待。

### 8.2 体验验收

1. 若后台正在进行较慢的模拟 step，创建/删除仍可能等待，但等待点应能解释为“命令串行化”，而不是“创建逻辑本身很重”。
2. 创建/删除成功后，前端不应因为不必要的全量重建而把 loading 时间继续拉长。

### 8.3 测试验收

建议补充或更新以下测试：

1. `useCreateAvatarPanel` 的命令后处理测试。
2. `useDeleteAvatarPanel` 的命令后处理测试。
3. `worldStore.fetchState()` 与 `applyStateSnapshot()` 的事件语义测试。
4. 若保留事件重载，补一个“普通创建/删除不会清空事件页”的回归测试。

## 9. 结论

删除角色确实和创建角色有同类问题，而且问题结构几乎一样。

所以这次修复不应该只盯着“创建慢”，而应该把它抽象成一条更稳的语义线：

1. 创建/删除是局部 mutation。
2. world state 是重建型同步。
3. 事件栏属于独立视图，不该被普通 mutation 的收尾动作误清空。

这样后续再碰到别的局部命令，也能沿着同一条原则往下长，不会越修越散。
