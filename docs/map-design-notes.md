# 地图设计思路

本文是官方地图扩展的简版设计说明，用来记录“为什么这样做”和“第一版应该守住什么边界”。具体实现细节见：

- `docs/specs/official-map-presets.md`

## 核心判断

地图第一版不做动态生成，也不做复杂地理沙盒。

当前更适合做成“官方地图预设 + 地图快照”：

1. 官方提供少数几张质量可控的地图。
2. 玩家开局选择地图。
3. 存档保存本局地图快照。
4. 后续如果角色行动改变地图，可以直接写入快照。

这样既能让世界有不同空间气质，又不会过早把系统拖进随机生成、地理模拟、经济网络和文化推导的大工程。

## 第一版地图目标

第一版只扩展地图形状和位置关系：

1. 从一张地图扩展到三张官方地图。
2. 三张地图统一尺寸。
3. 不新增 region。
4. 不新增地块类型。
5. 不新增风土文化字段。
6. 只改变 region 的形状、面积和位置。

也就是说，同一个 region id 在三张地图里仍然代表同一个区域，但它可以出现在不同位置，也可以有不同大小和边界形状。

## 三张官方地图

### 九州中土

默认均衡大陆。

适合新手和普通开局，地貌分布相对平均，城市、宗门、洞府和荒野不过度偏向某一种风格。

### 万山边境

山地、森林、险地更多。

它应该有更强的边境感和修行险境感：城市集中在谷地或山口，宗门和洞府更靠近山脉、密林、雪峰等位置。

### 沧海诸洲

水域、海岸、岛屿更多。

它应该让世界看起来更分散：region 分布在几个岛屿、半岛或海岸带上，城市多靠近海岸、内湾或河口。

## 地图预设与地图快照

`map_id` 表示本局开局选择了哪张官方地图。

`map_snapshot` 表示本局当前真实地图。

这两个概念要分开：

1. 新开局时，用 `map_id` 读取官方地图。
2. 游戏进行中，地图可能被未来系统改变。
3. 保存时，写入完整 `map_snapshot`。
4. 读档时，优先恢复 `map_snapshot`，而不是重新读取官方地图。

这样未来即使官方地图文件更新，旧存档也能保持当时的地图状态。

## 为什么保存全量地图

当前地图尺寸不大，全量保存地块矩阵和 region 归属矩阵成本很低。

第一版不做压缩，也不只保存差异。这样实现更直观，测试更简单，未来地图变化也更容易接入。

## 维护方式

地图不应该靠手工盲改 CSV。

维护流程应该是：

1. 编辑或生成地图 CSV。
2. 运行地图校验工具。
3. 生成 PNG 预览图。
4. 查看预览图修正空间结构。
5. 在前端实机验证地图渲染、标签、宗门层和角色位置。

地图是视觉资产，必须能看图调整。

## 后续方向

本次只做官方地图预设。后续可以继续扩展：

1. region 风土文化。
2. 角色出生地与地区气质。
3. 宗门所在地影响宗门风格。
4. 事件发生地影响故事文本。
5. 灾害、战争、宗门行动改变地图。
6. 外接控制 API 查询或修改地图状态。

这些都建立在第一版的 `map_id + map_snapshot` 基础上。
